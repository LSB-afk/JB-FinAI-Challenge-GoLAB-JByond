# 왜 3기능 모음이 아니라 하나의 금융 AX 콘솔인가

# One Console 논거: 3개 기능이 아니라 하나의 금융 AX 운영체계

## 결론

`JB LocalGuard OS`는 여신·전세·피싱 데모 3개를 나열한 앱이 아니라, 모든 금융 케이스를 같은 운영계약으로 처리하는 **금융 AI Agent 운영 콘솔**이다. 핵심은 기능 수가 아니라, `Case → AgentRun → Agent → Skill → Evidence → Approval → Audit` 한 벌이 모든 도메인과 계열사를 관통한다는 점이다. 이 구조는 `_canon.md` §0·§8의 제품 정의와 운영계약, `키스톤-확정`의 `[계열사 × 담당직군]` 축, D15의 승인·감사 설계 근거, D31의 실제 직군 seat 센서스와 정합한다.

## 1. 공통 운영계약이 모든 도메인을 관통한다

모든 업무는 먼저 `Case`가 된다. 전주 카페 여신심사(`JBG-104`, riskScore 88), 전세 위험, 보이스피싱, 사후관리 EWS는 서로 다른 화면 기능이 아니라 같은 상태기계 위의 다른 `caseType`이다.

| 계층 | 역할 | 공통성 |
|---|---|---|
| `Case` | 고객/거래/위험 신호를 하나의 업무 단위로 묶음 | 여신·전세·피싱·사후관리 모두 동일 |
| `AgentRun` | 특정 Agent가 케이스에 대해 실행한 판단·초안·검증 기록 | 실행 단위 표준화 |
| `Agent` | 14개 운영 에이전트 + 사람 승인자 2명 | canon §2 |
| `Skill` | 규정검색, 위험평가, 문서체크, 반출스캔 등 재사용 절차 | 도메인별 장착만 다름 |
| `Evidence` | 판단 근거·출처·불확실성·다음 확인 항목 | canon §8, D11 |
| `Approval` | L0~L4 승인 게이트 | RM·준법 승인자 2역할 |
| `Audit` | 변경 불가 감사 원장 | D15, canon §8 |

따라서 “전세 기능”, “피싱 기능”, “여신 기능”이 따로 있는 것이 아니라, **하나의 케이스 운영 커널에 도메인별 Agent/Skill/Evidence 묶음이 꽂히는 구조**다.

## 2. PII 4중방어와 승인 게이트는 전 도메인 공통 통제면이다

차별성은 데모별 AI 기능보다, 모든 기능이 같은 금융 통제면을 통과한다는 데 있다.

| 공통 통제 | 설계 |
|---|---|
| 데이터 등급제 | `restricted / confidential / internal / public`로 입력 분류 |
| 토큰화 | 원본 PII는 국내 PII 볼트/온프레 경로에만 보관 |
| 모델 라우팅 | 원본 PII 필요 작업은 로컬/온프레, 비식별 공개정보 분석만 외부 모델 |
| 반출스캔 + 감사원장 | 외부 전송 직전 DLP 스캔, 결과는 Audit에 기록 |

근거: `_canon.md` §0·§4는 “고객 원본 PII·신용정보는 절대 외부 반출 금지”와 4중방어를 고정한다. D15는 사람 승인과 감사 원장이 금융 AI에서 무단 실행 방지, 책임 추적, 복구 가능성에 강하다고 정리한다. 단, D15가 지적하듯 승인 버튼만으로는 부족하므로 근거 강제, 반대/재심, SLA, kill switch, immutable evidence가 같이 있어야 한다.

## 3. 역할축 콘솔은 담당자의 실제 업무 흐름을 따른다

키스톤은 `계열사(전북은행/JB우리캐피탈) × 담당직군`이다. 도메인은 별도 앱이 아니라 그 직군이 처리하는 케이스다.

D31 기준 실제 직군은 `RM · 여신심사 · 사후관리 · 준법 · AML`이다. `전세 담당`, `피싱 담당`을 별도 직군처럼 말하면 근거가 약하다. 전세는 RM/여신심사/준법이 보는 고위험 케이스이고, 피싱은 준법/AML이 보는 고위험 케이스다.

| 담당직군 | 콘솔에서 보는 케이스 |
|---|---|
| RM | 전주 카페 여신 상담, 정책금융 매칭, 고객 안내 초안 |
| 여신심사 | 상환능력, 담보, 전세 보증 가능성, 조건부 승인 |
| 사후관리 | EWS, 자금용도 점검, 연체/부실 징후 |
| 준법 | 고객 고지, 계약 문구, PII 반출, 설명가능성 |
| AML/FDS | 보이스피싱, 이상거래, 지급정지/해제 검토 |

즉 콘솔은 도메인 탭 모음이 아니라 “담당자가 하루 동안 처리하는 케이스 큐”다. 같은 사람이 여신 케이스를 보다가 전세 위험 케이스를 보고, 준법 승인 대기열과 Audit을 확인하는 흐름이 하나의 업무면 안에 있다.

## 4. 모듈식 확장은 같은 뼈대에 계열사·직군·도메인만 스왑한다

전북은행과 JB우리캐피탈은 같은 도메인을 그대로 복제하지 않는다. 전세는 은행 전용이고 캐피탈에는 부재한다는 점이 오히려 모듈식 확장성을 보여준다.

| 계열사 | 자연스러운 도메인 |
|---|---|
| 전북은행 | 여신, 전세보호, 보이스피싱, 사후관리 |
| JB우리캐피탈 | 오토/개인/기업/부동산금융 여신, 사후관리/EWS, AML |

같은 것은 `Case → ... → Audit`, PII 4중방어, 승인 게이트, Evidence/Audit 구조다. 다른 것은 계열사별 업무 모듈과 Agent/Skill 조합이다. 이것이 `_canon.md` §8·§9, `키스톤-확정`, D31의 확장 seat 논리와 맞다.

## 심사 5.1 차별성 문장

기존 금융 AI가 특정 업무 자동화나 챗봇에 머문다면, `JB LocalGuard OS`는 **업무 케이스, 전문 에이전트, 근거, 승인, 감사가 한 원장으로 연결되는 운영체계**다. 여신·전세·피싱은 별도 기능 3개가 아니라, 같은 금융 AX 커널이 서로 다른 직군과 계열사 업무에 적용되는 세 가지 증명 사례다.

## 반례 대응 문장

“그냥 기능 3개 아니냐”에 대한 답은 명확하다. 기능 3개라면 각 데모마다 데이터·승인·감사·보안 로직이 따로 있어야 한다. 그러나 본 설계는 모든 도메인이 동일한 `Case → AgentRun → Agent → Skill → Evidence → Approval → Audit` 계약, 동일한 PII 4중방어, 동일한 L0~L4 승인 게이트를 공유한다. 바뀌는 것은 도메인별 Agent와 Skill뿐이다. 그래서 이것은 3개 기능의 합이 아니라 하나의 금융 AX 콘솔이다.
