# JB우리캐피탈 전용 운영 하네스 (2026-07-03)

JB우리캐피탈 운영 포털은 메인(전북은행 Hero) 하네스와 **분리된 독립 하네스**다.
전략상 역할은 "그룹 확장성 증명 노드": 전북은행 Hero와 동일한 운영 패턴
(접수 → 분류 → 에이전트 → 사람 승인 → 감사 기록)이 계열사에 재사용됨을 보여준다.

## 1. 아키텍처

Presentation shell(4-zone 레이아웃, `styles.css`)은 공통 앱을 재사용하고,
JB우리캐피탈 business logic은 전용 파일에만 존재한다. 금지: 메인 하네스 alias,
label-only 재사용, affiliateId 없는 query.

| 파일 | 레이어 | 역할 |
| --- | --- | --- |
| `app/jbWooriCapitalSidebar.config.js` | config | route/nav/taxonomy/상태·필드 한국어 라벨 |
| `app/jbWooriCapitalAgents.registry.js` | registry | 13개 전용 에이전트 정의, 하네스 정책, 라우팅 규칙 |
| `app/wooricap-db.js` | repository | localStorage mock DB, seed, `jbwcRepository` 인터페이스 |
| `app/jbWooriCapitalServices.js` | service | count/search/triage/케이스 생성/에이전트 실행 기록 |
| `app/wooricap.helpers.js` | view-shared | 상태/pill/테이블/상세 패널/브레드크럼 공용 렌더 헬퍼 |
| `app/wooricap.view.board.js` | view | 업무 보드(대시보드) |
| `app/wooricap.view.cases.js` | view | 케이스/도메인/운영 현황 목록 뷰 모음 |
| `app/wooricap.view.wizard.js` | view | 신규 운영 건 접수 위저드(5단계) + 폼 바인딩 |
| `app/wooricap.view.harness.js` | view | 운영 에이전트 하네스 + 샘플 실행 바인딩 |
| `app/wooricap.sidebar.js` | binding | 사이드바 점유/복원, 내비, 검색, 카운트 |
| `app/wooricap.js` | router | 렌더러 조립, hash 라우팅, 이벤트 바인딩 |
| `app/wooricap.css` | style | `jbwc-` 스코프 전용 스타일(JBFG 디자인 오버레이) |

스크립트 로드 순서(의존성): config → registry → db → services → helpers → views → sidebar → router → app.js.

## 3. Sidebar count query mapping

모든 count는 `getJbWooriCapitalSidebarCounts()`에서 계산하며, 내부적으로
`jbwcTable(table, JBWC_AFFILIATE_ID)`만 사용한다(스코프 필수, 미지정 시 예외).

| countKey | 테이블 | 조건 |
| --- | --- | --- |
| board | ops_tasks | status ∈ {open, inProgress, overdue} ∪ dueAt=오늘 |
| approvals | approvals | status = pending |
| auditLogs | audit_logs | reviewRequired = true |
| privacyPermissions | privacy_permission_checks | status ∈ {open, needsReview} |
| integrations | external_connectors | health ∈ {degraded, down} ∪ status ∈ {pending, error} |
| cases | ops_cases | status ∈ JBWC_ACTIVE_CASE_STATUSES |
| personalFinance …complianceInternalControl | ops_cases | domain=해당 도메인 ∧ active |
| documents | document_cases | status ∈ {pending, needsReview} |
| vehicleLifecycle | vehicle_lifecycle_tasks | status ∈ {open, inProgress, overdue} |
| consumerProtection | consumer_protection_reviews | status ∈ {open, needsReview} |
| fds | fds_alerts | status ∈ {open, investigating, escalated} |
| urgentAlerts | fds_alerts+customer_support_cases | severity high/critical 미해결 + priority urgent 미해결 |
| complaints | customer_support_cases | status ∈ {open, inProgress, needsReview} |
| aiAnalysis | ai_analysis_requests | status ∈ {queued, running} |
| aiAssist | ai_recommendations | status ∈ {active, proposed} |
| agentHarness | agent_runs+harness_agents | run: {queued,running,needsReview} + agent: {needsReview,escalated} |
| capabilities | business_capabilities | status ∈ {enabled, proposed} |
| roles | role_assignments | status ∈ {active, needsReview} ∪ reviewRequired |
| inspections | inspection_schedules | status ∈ {upcoming, overdue} |

## 5. Agent registry와 handoff rule

13개 에이전트(`jbWooriCapitalAgents`): 오케스트레이터 1 + 도메인 11 + 운영지표 QA 1.
각 항목은 id/agentKey/name(영문 식별)/displayName(한국어 표기)/책임/입출력 스키마/
allowed·blockedActions/dbReads·dbWrites/handoffRules/guardrails/metrics를 갖는다.

라우팅: `routeJbWooriCapitalCase(input)` → `jbWooriCapitalRoutingRules[domain]`.

핸드오프 규칙(트리거 → 대상):
- 문서/담보 도메인 또는 첨부 존재 → `jbwc-doc` (문서 상태 확인, 초기상태 waitingDocuments)
- 차량 도메인 또는 vehicleRefId → `jbwc-vehicle` (초기상태 waitingVehicleTask)
- 소비자보호 도메인 또는 민원/권리구제/금리인하/청약철회/위법계약/자료열람 키워드 → `jbwc-protect` (사람 검토 필수, pendingCustomerProtectionReview)
- FDS 도메인 또는 보이스피싱/피싱/이상거래 키워드 → `jbwc-fds` (에스컬레이션 필수, riskLevel ≥ high, pendingFdsEscalation, handoff status=escalated)
- 내부통제 도메인 또는 권한/개인정보/추심/약관/정책 키워드 → `jbwc-compliance` (사람 검토)
- riskLevel high/critical → requiresHumanReview=true, 미지정 시 pendingApproval
- priority urgent → `jbwc-metrics` (SLA 임박 기록)

실행 기록: `recordJbWooriCapitalAgentRun()` → agent_runs + (핸드오프별) agent_handoffs + audit_logs.

## 6. Guardrails

하네스 정책(`jbWooriCapitalOpsHarness.policy`) + 에이전트 공통 blockedActions:
- 실제 대출 승인/거절·금리/한도 산정·신용평가 금지
- 실제 개인정보 원문 조회/저장/전송 금지 (익명 참조 ID만: CUST-/CONTRACT-/VEH-)
- 실제 계좌/결제/자동이체 변경·전자약정 체결 금지
- FDS·보이스피싱 high/critical 자동 종결 금지 — 사람 에스컬레이션 필수
  (e2e가 "high/critical run은 completed/closed 불가" 불변식을 검증)
- 소비자보호·법규·권리구제 답변은 담당자 검토 필요 (requiresHumanReview 고정 true)
- 모든 AI output은 "내부 운영 참고용" — 화면마다 모의(mock) 데이터 고지 유지

## 8. 운영 DB 전환 준비 (repository interface)

`wooricap-db.js`의 `jbwcRepository`가 유일한 저장소 계약이다:

```js
jbwcRepository = {
  table(name, affiliateId),  // affiliateId 필수 — 미지정 시 예외
  insert(name, row),         // row.affiliateId 채워진 상태로만 호출
  nextId(prefix, name),      // 데모용 순번 — 운영 전환 시 시퀀스/UUID
  reset(),                   // 데모 seed 재적재 — 운영 전환 시 제거
  snapshot(),                // mock 원본 접근 — 운영 전환 시 제거
}
```

전환 시 주의사항:
- **환경변수**: DB 접속 정보는 서버 사이드 환경변수로만 주입한다. 프런트 코드/저장소에
  운영 credential 하드코딩 금지(현재 코드에는 어떤 credential도 없음).
- **권한**: affiliateId 스코프를 클라이언트 필터가 아닌 **서버 권한(행 수준 보안/뷰)**으로
  강제해야 한다. 현재 `affiliateId scope is required` 예외는 계약의 문서화일 뿐
  보안 경계가 아니다.
- **마이그레이션**: 테이블 스키마는 §7의 seed 구조를 따른다. version 필드(현재 3)를
  마이그레이션 버전으로 승계하고, localStorage → 서버 이전 시 데모 데이터는 이관하지
  않는다(전부 mock).
- **금지**: 실제 고객 개인정보/신용정보/대출 실행/금융거래 시스템과의 연결.
  외부 실데이터 증명은 공공 API 1종(국토부/ECOS 후보)만 별도 검토.

## 9. 검증 방법

```bash
npm run build      # verify_static.py — 파일/문자열/문법 계약 (46개 파일)
npm test           # 동일
npm run test:e2e   # localguard.spec.js(23) + wooricap.spec.js(11)
```

JBWC 전용 e2e(`tests/e2e/wooricap.spec.js`): route persistence,
affiliate scoping 격리(타 계열사 seed 미노출), 금지 메뉴/alias 부재,
도메인별 케이스 생성 기록(fds_alerts/consumer_protection_reviews/
vehicle_lifecycle_tasks/document_cases), 하네스 실행·핸드오프 규칙,
고위험 자동 종결 금지 불변식, 데스크톱/태블릿/모바일 스크린샷 스모크
(`test-results/screenshots/jbwc-*.png`).
