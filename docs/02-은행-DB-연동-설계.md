# 기존 은행 DB 연동 설계 (명문화)

> 목적: 현재 localStorage mock 저장소로 동작하는 프로토타입을, 기존 은행 내부 시스템(계정계·정보계·채널계)과
> **어떤 경계·순서·통제로** 연결할 수 있는지 명문화한다.
> 대전제: **실제 고객 개인정보/신용정보 원문, 대출 실행, 금리/한도 산정, 금융거래 실행은 어떤 단계에서도 연결하지 않는다.**

## 1. 현재 상태 (Phase 0 — 검증된 기준선)

- 저장소: 브라우저 localStorage (`jbwc-ops-db-v3`, `jb-finance-support-state-v4`)
- 접근 계약: 화면·서비스 레이어는 저장소를 직접 만지지 않고 **repository 인터페이스만** 사용한다.

```js
// app/wooricap-db.js — 유일한 저장소 계약
jbwcRepository = {
  table(name, affiliateId),  // affiliateId 필수 — 미지정 시 예외 발생
  insert(name, row),         // row.affiliateId가 채워진 상태로만 호출
  nextId(prefix, name),      // 데모용 순번 ID
  reset(),                   // 데모 seed 재적재 (운영 전환 시 제거)
  snapshot(),                // mock 원본 접근 (운영 전환 시 제거)
}
```

- 서버로 그대로 승격 가능한 함수 경계(1:1 API 후보): `computeRiskDecision`,
  `buildDashboardData`, `auditChainRecords`, `moveCaseToColumn`,
  `getJbWooriCapitalSidebarCounts`, `searchJbWooriCapitalRecords`,
  `createJbWooriCapitalOpsCase`, `recordJbWooriCapitalAgentRun`, `routeJbWooriCapitalCase`
- 이 경계는 e2e 33개 시나리오로 고정되어 있어, 구현체를 교체해도 동작 동일성을 회귀 검증할 수 있다.

핵심 결정:

1. **콘솔은 은행 DB에 직접 붙지 않는다.** 반드시 신설 서비스 계층(BFF)을 경유한다. 프런트에는 접속 정보·credential이 존재하지 않는다.
2. **계정계 직접 연결 금지.** 조회가 필요한 데이터는 이미 은행이 운영 중인 정보계(ODS/DW) 읽기 전용 복제본에서 가져온다. 계정계 반영은 기존 배치/CDC 경로를 그대로 쓴다.
3. **운영 워크플로 데이터(케이스·승인·감사·에이전트 실행)는 신설 운영 DB에 저장**한다. 은행 원장 데이터와 물리적으로 분리해 장애·보안 영향 범위를 차단한다.
4. **PII는 토큰화 참조로만.** 콘솔·운영 DB·LLM에는 `CUST-*` 같은 비식별 참조 ID만 흐른다. 원문 조회가 필요한 업무는 기존 은행 단말(권한 통제된 화면)에서 수행하고, 본 시스템은 참조 ID로 연결만 한다.

## 3. 단계별 전환 로드맵

| Phase | 내용 | 데이터 방향 | 선행 조건 |
| --- | --- | --- | --- |
| **0. 현재** | localStorage mock. 데모·UX·계약 검증 | 없음 (브라우저 내) | 완료 — e2e 33개 |
| **1. 읽기 전용 연동** | repository의 `table()`만 BFF fetch로 교체. 정보계 ODS 복제본·FDS 경보 피드를 read-only로 표시 | 은행 → 콘솔 (단방향) | SSO·RBAC, 토큰화 서비스, 망내 배포 |
| **2. 운영 워크플로 저장** | `insert()` 계열(케이스 접수·승인·감사·agent_runs)을 운영 DB에 저장. 전자결재 연동으로 승인 위임 | 콘솔 ↔ 운영 DB | 운영 DB 스키마 마이그레이션, 감사 로그 불변화(WORM/서명) |
| **3. 이벤트 연동** | FDS 경보·문서 접수 이벤트를 MQ 구독으로 실시간 수신, 케이스 자동 생성(단, 자동 종결은 계속 금지) | 은행 → 콘솔 (이벤트) | 이벤트 스키마 합의, 재처리/중복 제거 설계 |
| **금지 (모든 Phase)** | 계정계 직접 접속, 대출 실행/금리·한도 산정/신용평가 API 호출, PII 원문 저장·전송, 고객 대상 자동 발송 | — | — |

Phase 1이 끝나면 콘솔 코드는 **어댑터 파일 1개 교체** 외에 변하지 않는 것이 목표다
(repository 인터페이스가 그 경계이며, e2e가 동작 동일성을 보증한다).

## 5. 스코프·보안 통제 (must)

1. **행 수준 보안(Row-Level Security).** 현재 `affiliateId scope is required` 예외는 계약의 문서화일 뿐 보안 경계가 아니다. 운영 전환 시 DB RLS 또는 스키마 분리로 **서버에서** 계열사 스코프를 강제한다. (검증 기준: 프로토타입 e2e의 "타 계열사 seed 미노출" 시나리오를 서버 대상 통합 테스트로 승격)
2. **인증/권한.** 행내 SSO(사번) + RBAC. 역할(RM/준법/FDS/소비자보호/내부통제)별 화면·행동 권한은 현재 `role_assignments` 모델을 IAM 클레임으로 대체.
3. **credential 관리.** DB/API 접속 정보는 서버 환경변수 또는 시크릿 매니저로만 주입. 저장소·프런트 코드에 하드코딩 금지(현재 코드에 credential 없음 — 유지).
4. **LLM 데이터 경계.** 외부 LLM 호출 전 게이트웨이에서 PII 패턴 차단(비식별 검증). PII가 필요한 스텝은 내부망 로컬 모델(EXAONE 옵션) 또는 규칙 기반 시뮬레이션으로 처리. 모델 부재 시에도 업무가 완주되는 fallback 유지.
5. **감사 불변성.** 자체 해시 체인(`auditChainRecords`)을 유지하되, 운영 전환 시 서명·타임스탬프 서버 또는 WORM 스토리지로 보강하고 은행 감사계에 사본 전송.
6. **망 구성.** 콘솔·BFF·운영 DB는 내부망 배치. 외부 LLM 사용 시 별도 아웃바운드 프록시 + 전송 데이터 로깅.
7. **관련 규제 관점.** 개인정보보호법·신용정보법(가명처리/목적 제한), 전자금융감독규정(망분리·접근통제) 준수를 전제로 설계 검토 필요 — 구체 조항 매핑은 준법 부서 검토 사항으로 남긴다.

## 6. 마이그레이션 체크리스트

- [ ] 운영 DB 스키마 생성 — `docs/03-JB우리캐피탈-하네스.md` §7 seed 구조를 DDL로 승격 (version=3를 마이그레이션 버전으로 승계)
- [ ] ID 채번: `jbwcNextId` 순번 → DB 시퀀스/UUID 교체
- [ ] 데모 seed는 **이관하지 않는다** (전부 mock). 초기 데이터는 정보계 복제본에서 적재
- [ ] repository 어댑터 작성: `table/insert` → BFF REST 호출 (낙관적 UI 유지 여부 결정)
- [ ] `reset()`/`snapshot()` 제거 및 데모 전용 플래그 분리
- [ ] e2e 재사용: Playwright 스위트를 스테이징 BFF 대상으로 실행하는 프로파일 추가
- [ ] 환경변수 목록 확정: `DB_URL`(서버 전용), `PUBLIC_API_KEY`(공공 API, 서버 전용), `LLM_GATEWAY_URL`, `SSO_ISSUER` 등 — 프런트 노출 변수 없음
- [ ] 부하·장애 시나리오: 정보계 복제 지연 시 콘솔은 `lastSyncAt`/stale 표시로 강등 동작 (현재 UI 상태 모델 재사용)

## 7. 요약

현재 프로토타입은 **저장소 계약(repository interface)과 검증 스위트(e2e 33개)** 를 이미 갖추고 있어,
은행 연동은 "콘솔 재작성"이 아니라 **어댑터 교체 + 서버 계층 신설** 문제로 축소된다.
연결 순서는 읽기 전용(정보계 복제본) → 운영 워크플로 저장 → 이벤트 구독이며,
계정계 직접 연결·PII 원문·금융거래 실행은 모든 단계에서 금지된다.
